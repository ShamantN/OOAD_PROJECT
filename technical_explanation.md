# E-Commerce Nexus — Complete Project Technical Explanation

---

## 1. The Big Picture — What is E-Commerce Nexus?

It is a **full-stack web application** that simulates how a real e-commerce platform works internally. Think of it as a simplified Flipkart backend. It has three people using it:
- A **Customer** who browses products, adds to cart, pays, and can cancel
- An **Admin** who creates products, manages stock, and ships/delivers orders
- The **System** itself which enforces business rules automatically

The entire project is split into two completely independent parts — a **Java backend** and a **JavaScript frontend** — that talk to each other only through HTTP requests over a REST API.

## 2. Architectural Pattern — Layered MVC

**MVC stands for Model-View-Controller.** It is a design pattern that separates an application into three responsibilities so that no single class is doing too many things.

Your project implements a **Strict Layered MVC**, which means the layers can only talk to the layer directly below them — like a chain of command:

```
Browser (View)
    ↕ HTTP/JSON
Controller Layer      ← handles HTTP requests/responses only
    ↕ Java method calls
Service Layer         ← contains all business logic
    ↕ Java method calls
Repository Layer      ← communicates with the database only
    ↕ SQL (generated by JPA)
Database (MySQL)
```

### The View
Your HTML/CSS/JS files are the View. They display data and capture user inputs. They have no business logic — they just call the backend API and render whatever comes back.

### The Controller
Files like `OrderController.java`, `AdminController.java`, `UserController.java`. Their **only job** is to receive an HTTP request, extract the data from it (URL path, request body, headers), hand it to the Service layer, and send back the HTTP response. Controllers never touch the database directly.

### The Service
Files like `OrderService.java`, `PaymentService.java`, `FulfillmentService.java`. This is the brain. All business rules live here — checking stock, calculating totals, enforcing state machine rules, applying cancellation penalties. Services never know anything about HTTP.

### The Repository
Files like `OrderRepository.java`, `ProductRepository.java`. Their only job is to talk to the database — find records, save records, run queries. They use Spring Data JPA so most of the time you don't even write SQL — Spring generates it for you from method names.

### The Model
Files like `Order.java`, `Product.java`, `User.java`, `Inventory.java`. These are plain Java classes annotated with `@Entity`, which tells Hibernate to map them to database tables. Each field becomes a column, each object becomes a row.

## 3. Spring Boot — What it Does for You

Spring Boot is a framework that removes enormous amounts of setup work. Here is what it provides in your project:

### Dependency Injection (`@Autowired`)
Without Spring, if `OrderController` needs `OrderService`, you'd manually write `OrderService s = new OrderService()`. With Spring, you just write `@Autowired private OrderService orderService` and Spring creates and injects the object automatically. This is called the **IoC (Inversion of Control) Container** — Spring controls when objects are created, not you.

### Auto-configuration
Because you added `spring-boot-starter-web` in your `pom.xml`, Spring Boot automatically sets up:
- An embedded **Apache Tomcat** server running on port 8080
- A **Jackson** JSON serializer/deserializer so Java objects automatically convert to JSON for the frontend
- **CORS configuration** support

Because you added `spring-boot-starter-data-jpa` + the MySQL driver, Spring Boot automatically:
- Creates a connection pool to your MySQL database
- Runs **Hibernate** which reads your `@Entity` classes and auto-generates the SQL schema (DDL)

### `@RestController` and `@RequestMapping`
These annotations tell Spring Boot which Java class handles which URL. For example:
```java
@RestController
@RequestMapping("/api/orders")
public class OrderController {
    @PostMapping("/place")
    public ResponseEntity<Order> placeOrder(...) { ... }
}
```
This means: when the browser sends `POST http://localhost:8080/api/orders/place`, Spring routes it to this method automatically.

### `@Transactional`
Wraps a method in a database transaction. If any operation inside fails, everything rolls back. Critical for your `OrderService.processNewOrder()` — if the second item in a cart is out of stock, the stock reduction for the first item is automatically reverted.

## 4. The Database & ORM (Object-Relational Mapping)

Your database has these tables, auto-created by Hibernate from your Java classes:

| Table | Java Entity | Purpose |
|---|---|---|
| `users` | `User.java` | Stores name, email, password, role |
| `products` | `Product.java` | Stores product name and price |
| `warehouse` | `Warehouse.java` | Stores warehouse location |
| `inventory` | `Inventory.java` | Links product + warehouse + stock count |
| `orders` | `Order.java` | Stores order total, status, which user |
| `order_items` | `OrderItem.java` | Each product in an order, with quantity |
| `payments` | `Payment.java` | 1-to-1 with order, stores payment status |
| `cancellation_impacts` | `CancellationImpact.java` | Financial penalty record for cancelled orders |

### Relationships — How Tables Connect
- `Order` has a `@ManyToOne` to `User` — many orders can belong to one user
- `Order` has a `@OneToMany` to `OrderItem` — one order has many line items
- `OrderItem` has a `@ManyToOne` to `Product` — many items can reference the same product
- `Payment` has a `@OneToOne` to `Order` — every order has exactly one payment record
- `Inventory` has a `@ManyToOne` to both `Product` and `Warehouse`

In Java these look like:
```java
@ManyToOne
@JoinColumn(name = "user_id")
private User user;
```
Hibernate translates this into a `user_id` foreign key column in the `orders` table in MySQL.

## 5. REST API — How the Frontend and Backend Talk

REST means every URL represents a **resource** and HTTP verbs (GET, POST, PUT, DELETE) represent what action you're doing with it.

### Your API Endpoints

| Method | URL | Who calls it | What it does |
|---|---|---|---|
| POST | `/api/users/register` | Frontend | Creates a new user account |
| POST | `/api/users/login` | Frontend | Validates credentials, returns user data |
| GET | `/api/catalog/products` | Frontend | Returns all in-stock products (via DTO) |
| POST | `/api/orders/place` | Frontend | Places an order, deducts inventory |
| GET | `/api/orders/user/{id}` | Frontend | Returns order history for a customer |
| POST | `/api/orders/{id}/pay` | Frontend | Transitions order from CREATED → PAID |
| POST | `/api/orders/{id}/cancel` | Frontend | Cancels order, restores stock, calculates penalty |
| POST | `/api/admin/products` | Admin frontend | Creates a product (RBAC protected) |
| PUT | `/api/admin/inventory/restock` | Admin frontend | Adds stock to warehouse (RBAC protected) |
| GET | `/api/admin/orders` | Admin frontend | Returns all orders in system (RBAC protected) |
| PUT | `/api/admin/orders/{id}/ship` | Admin frontend | Transitions order PAID → SHIPPED (RBAC) |
| PUT | `/api/admin/orders/{id}/deliver` | Admin frontend | Transitions SHIPPED → DELIVERED (RBAC) |
| GET | `/api/admin/metrics/cancellations` | Admin frontend | Returns aggregated financial loss metrics (RBAC) |

## 6. OOAD Concepts Applied

### 6.1 Finite State Machine

An order cannot jump from `CREATED` directly to `DELIVERED`. It must follow this exact chain:

```
CREATED → PAID → SHIPPED → DELIVERED
   ↓        ↓       ↓
         CANCELLED (not allowed from DELIVERED)
```

This is enforced in two places:
- `FulfillmentService.java` — throws `RuntimeException("Order is not in PAID status")` if you try to ship an unsupported state
- `OrderStatus.java` — stores the state as a database ENUM so it is permanent in MySQL

The payment also has its own state machine: `PaymentStatus.FAILED` (default) → `PaymentStatus.SUCCESS` (after payment).

### 6.2 Encapsulation via DTOs

`Product.java` is your internal database entity. Instead of returning the entity directly, your `CatalogController` returns a `ProductCatalogDTO`:

```java
public class ProductCatalogDTO {
    private int productId;
    private String name;
    private double price;
    private int stock;  // aggregated from ALL warehouses
}
```

The JPQL query that populates this does several things simultaneously:
- JOINs the `Inventory` and `Product` tables
- GROUPs by product
- SUMs the total stock across all warehouses
- FILTERs out products with zero stock (`HAVING SUM(stock) > 0`)

All in one database call. The Frontend never sees raw inventory or warehouse data.

### 6.3 Separation of Concerns

Every class has exactly one job. Example — the `placeOrder` flow:
- `OrderController` → reads the HTTP request body, calls `orderService.processNewOrder()`, returns HTTP 201
- `OrderService` → validates stock, calculates total, links items, saves order, saves payment. Knows nothing about HTTP.
- `OrderRepository` → saves the order to the database. Knows nothing about business rules.
- `InventoryRepository` → reduces stock count. Knows nothing about orders.

### 6.4 Role-Based Access Control (RBAC)

The `AdminController` has a private guard method:
```java
private void verifyAdminAccess(int adminId) {
    User user = userRepository.findById(adminId)
        .orElseThrow(() -> new ResponseStatusException(403, "Access Denied"));
    if (user.getRole() != Role.ADMIN) {
        throw new ResponseStatusException(403, "Insufficient permissions");
    }
}
```
Every admin endpoint calls this first. A customer's user ID will be rejected because their role is `CUSTOMER`, not `ADMIN`.

### 6.5 Dynamic Penalty Logic (Business Intelligence)

When cancelling a SHIPPED order, `OrderService.cancelOrder()` applies:
- **15% cancellation fee** = `order.getTotalAmount() * 0.15`
- **$10 flat delivery charge loss**
- **Final refund** = `totalAmount - cancellationFee - deliveryChargeLoss`

This is saved as a `CancellationImpact` entity. The repository uses JPQL aggregate queries:
```java
@Query("SELECT COALESCE(SUM(c.deliveryChargeLoss), 0) FROM CancellationImpact c")
double sumTotalDeliveryChargeLoss();
```
The Admin dashboard calls these on page load and shows the total financial damage in real time.

## 7. The Frontend — How It Works

The frontend has three pages:

### `login.html` + `auth.js`
- Handles Registration and Login forms
- On success, stores the user object in `localStorage` (browser's local key-value store)
- Reads the `role` from the response and redirects: ADMIN → `admin.html`, CUSTOMER → `index.html`

### `index.html` + `catalog.js`
- On load, reads user from `localStorage` — if missing, redirects to login
- Fetches `/api/catalog/products` and renders product cards dynamically
- Shopping cart stored in `localStorage` as a JSON array
- "Checkout" builds the multi-item JSON payload and sends it as `POST /api/orders/place`
- "My Orders" section fetches `/api/orders/user/{id}` and renders order rows with appropriate action buttons per state

### `admin.html` + `admin.js`
- On load, checks role — if not ADMIN, blocks the page and redirects
- `adminFetch()` wrapper automatically adds `Admin-User-Id` header to every request
- Renders the Order Fulfillment table by fetching `GET /api/admin/orders`
- Ship/Deliver buttons appear dynamically based on the order's current state
- Financial Impact Dashboard fetches `GET /api/admin/metrics/cancellations` on page load

### Error Handling
Every `fetch()` call uses a `safeJson()` helper that checks the `Content-Type` header before calling `.json()`. If the backend returns a plain error string, the UI handles it gracefully instead of crashing with a `JSON.parse error`.

## 8. CORS Configuration

Browsers block JavaScript from calling APIs on different domains/ports by default. Since your frontend runs on port 5500 and backend on port 8080, this would be blocked. Your `CorsConfig.java` tells Spring Boot to explicitly allow:
- **Origins:** `http://localhost:5500` and `http://127.0.0.1:5500`
- **Methods:** GET, POST, PUT, DELETE
- **Headers:** All headers (needed for the `Admin-User-Id` custom header)

## 9. How Data Flows — Complete Example

**Customer buys a Laptop:**

1. Browser sends: `POST /api/orders/place` with JSON body:
   ```json
   { "user": {"userId": 3}, "items": [{"product": {"productId": 1}, "quantity": 2}] }
   ```

2. `OrderController` receives it, deserializes JSON to an `Order` object, calls `orderService.processNewOrder(order)`

3. `OrderService` opens a `@Transactional` block:
   - Fetches real `Product` from DB using `productId` (ignores client-sent price)
   - Finds `Inventory` for that product
   - Checks `inventory.isAvailable(2)` — throws exception if stock < 2
   - Calls `inventory.reduceStock(2)` — saves to DB
   - Sets `item.setPrice(realProduct.getPrice())` — uses backend price, not frontend price
   - Calculates total = price × quantity
   - Sets order status to `CREATED`, saves order
   - Creates a `Payment` record with status `FAILED`, links it to the order

4. Returns the saved `Order` object as JSON to the browser

5. `catalog.js` receives it, shows success message with order ID, clears the cart, refreshes My Orders
